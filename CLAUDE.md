## Commands

- `pnpm dev` — start Vite dev server
- `pnpm build` — typecheck (`tsc -b`) then production build via Vite
- `pnpm lint` — oxlint
- `pnpm preview` — preview a production build

No test suite exists in this repo.

Package manager is pnpm — a `pnpm-workspace.yaml` exists (not a real monorepo; it holds `packages: ['.']` plus `allowBuilds` for `core-js`/`es5-ext`/`esbuild` postinstall scripts). This file is load-bearing: pnpm 11 does **not** honor `pnpm.onlyBuiltDependencies` in `package.json` — the working mechanism is `allowBuilds` in `pnpm-workspace.yaml`. Don't delete this file or move that config back to `package.json` expecting it to work (confirmed empirically — see git history around "fix build workers").

Deploys to Cloudflare Workers via `wrangler.jsonc` (static assets from `./dist`, SPA fallback via `not_found_handling: "single-page-application"`, routed at `auto-reader.naofalleoagusta.com/*`). To validate wrangler config changes without a real deploy: `npx wrangler deploy --dry-run`.

## Architecture

Local-first EPUB/PDF reader: Vite + React 19 + TypeScript + Tailwind v4 (CSS-native `@theme`, no `tailwind.config.js`) + zustand. No backend for the reader itself — everything happens client-side from a `File` the user drops or picks; the only persistence is the browser's own IndexedDB/localStorage.

**Two separate HTML entries** (`vite.config.ts` → `build.rollupOptions.input`): root `index.html` is a plain static marketing/landing page (no React, no framework — a few lines of inline vanilla JS for one interactive demo) styled via `src/landing.css`. `app/index.html` is the actual React SPA (unchanged behavior from before the split — `/src/main.tsx` is an absolute path, so relocating the HTML file didn't require touching it). This split exists so the landing page ships **zero** of the app's JS (react/zustand/epubjs/pdfjs-dist — ~950KB gzipped) and reads as real static HTML to crawlers. Don't merge these back into one entry without re-deriving why they're split.

**State**: `src/state/useReaderState.ts` is a single zustand store (wrapped in `persist` middleware) holding the loaded `Book`, reading `position` (chapter/block index), `currentWordIndex`, playback state (`isReading`, `readingSpeedWpm`, `isSpeechEnabled`), font/theme settings, UI toggles, and the in-memory `library` cache. Only `font`/`theme`/`readingSpeedWpm`/`isSpeechEnabled`/`lastOpenedBookId` are persisted to localStorage (`partialize`) — actual book content lives in IndexedDB (`src/lib/db.ts`), never localStorage (way over its quota for real book text). All components read via selectors; only `App.tsx` wires the behavior hooks together and passes props down — components never touch the store directly.

**Persistence** (`src/lib/db.ts`): two IndexedDB object stores — `books` (full parsed `Book`, keyed by id) and `library` (lightweight `LibraryEntry` records: metadata + `lastPosition` + `lastWordIndex`, cheap to list for the sidebar without loading book text). The DB connection is cached at module level (`openDb()` reuses one promise, doesn't reopen per call) and `updateLastPosition` does its read-modify-write in a single transaction. Re-uploading the same file dedupes by filename+size (`findLibraryEntryByFile` — a heuristic, not a content hash) and resumes the saved position instead of re-parsing/restarting. `App.tsx` flushes the exact current word position on `visibilitychange`/`pagehide` since `setWordIndex`'s IndexedDB write is throttled (~1/sec) during playback.

**Parser seam** (`src/lib/parsers/`): `resolveParser(file)` in `index.ts` dispatches to `epubParser.ts` or `pdfParser.ts` by extension. Both implement the `BookParser` interface (`src/types/parser.ts`: `supports()` + `parse(file, onProgress?)`) and produce the same canonical `Book → Chapter → TextBlock` shape (`src/types/book.ts`). This interface is the reason swapping parsing strategy never touches `useBookParser.ts` or any component.
- `epubParser.ts` uses `epubjs` directly — walks `book.spine.spineItems` (not `book.loaded.spine`, which resolves to the wrong shape despite what epubjs's own `.d.ts` implies), extracts headings/paragraphs/blockquotes from each section's DOM, maps chapter titles from `book.navigation.toc`. epubjs's shipped types are unreliable in a few places (`Section.load()` is typed as sync-returning-`Document` but is actually async; `Spine.spineItems` isn't in the `.d.ts` at all) — the code works around this with narrow local type shapes rather than trusting the package's `.d.ts`.
- `pdfParser.ts` uses `pdfjs-dist`. The worker is wired via Vite's `?url` import (`import workerSrc from 'pdfjs-dist/build/pdf.worker.min.mjs?url'` → `GlobalWorkerOptions.workerSrc = workerSrc`) — no `vite.config.ts` changes needed for this. Chapters come from the PDF's outline/bookmarks when present and resolvable (top-level entries only, mapped to page ranges via `getDestination`/`getPageIndex`), falling back to one whole-document chapter when there's no usable outline. Paragraph boundaries are inferred from `pdf.js`'s per-line `hasEOL` signal plus a line-gap heuristic (`pdfLayout.ts`) — PDFs have no real semantic structure, so all PDF-derived blocks are `kind: 'paragraph'` (no heading/quote detection).
- Both parsers throw rather than silently return an empty book when there's no extractable text (e.g. scanned/image-only PDFs) — preserve that behavior in any new parser.

**Auto-advance vs. narration pacing, and word-level highlighting** (the trickiest runtime behavior — read before touching `useAutoAdvance.ts` or `useTextToSpeech.ts`):
- `useAutoAdvance.ts` paces silent reading via `setTimeout` (not `requestAnimationFrame` — rAF fully *stops* firing when the tab is backgrounded/phone locked; setTimeout only gets throttled, which is the actual desired behavior here — "keep reading while backgrounded" was an explicit requirement). It subdivides each block's estimated duration (`wordsToDurationMs` in `lib/wpm.ts`) evenly per word to drive `currentWordIndex`, and tracks elapsed time in a ref that survives a pure pause (so resume continues from the same word instead of restarting the paragraph).
- `useTextToSpeech.ts` is the pacing driver instead whenever narration is on — it advances on the browser's actual `utterance.onend`/`onerror`/`onboundary` events, not a word-count estimate (real speech duration doesn't match a guess closely enough: long paragraphs would leave dead silence, short ones would get cut off). Pausing calls native `speechSynthesis.pause()`/`.resume()` (not cancel+rebuild) so resume continues from the exact word. Has a safety-net timeout (word-count estimate × 4) in case `onend` never fires (no voices installed), and defers `speak()` by one tick after `cancel()` — calling them same-tick is a real Chrome/WebKit bug that stalls the speech queue for seconds.
- `useAutoAdvance` explicitly no-ops when `isSpeechEnabled` is true so the two never both drive `nextBlock()`.
- **Race condition both hooks guard against**: a manual navigation (e.g. clicking a paragraph) can happen just as an already-scheduled tick/utterance callback is about to fire for the *old* position. Since `nextBlock()` always reads live state, that stale callback would advance one past wherever the user just clicked. Both hooks' advance callbacks re-check that the position they were scheduled for still matches the store before calling `nextBlock()` — don't remove this guard when refactoring either hook.
- Paragraph blocks use `onPointerDown` rather than `onClick` (`ReaderCanvas.tsx`) — while playing, the word-follow auto-scroll can shift layout between press and release, so a release-based click can land on the next block down.

**Rendering perf**: `ReaderCanvas.tsx`'s per-paragraph rendering is a memoized `BlockView` (`React.memo`), and `Sidebar`/`ControlBar`/`CommandPalette` are also `React.memo`'d with their callback props stabilized via `useCallback` in `App.tsx` (and `chapterProgress`/`blockProgress` via `useMemo`). This matters because `currentWordIndex` changes very frequently during playback (every ~120ms in silent mode, every spoken word in narrated mode) — without this, every tick would re-render the entire chapter's block list plus the whole sidebar/control-bar/palette tree. Non-active blocks are passed a constant `currentWordIndex={-1}` (not the live value) specifically so their memo comparison never sees a "changed" prop from the tick. Keep new props flowing into these components stable (`useCallback`/`useMemo`) or the memoization silently stops working.

**Background playback**: `useWakeLock.ts` holds the screen awake while `isReading` (reacquires on `visibilitychange`, since the Wake Lock API auto-releases when hidden and doesn't reacquire itself) — only prevents *automatic* screen-lock from inactivity, not a manual lock-button press. `useMediaSession.ts` sets lock-screen metadata/transport controls — best-effort, mainly helps Chrome/Android keep the tab treated as active media; iOS Safari still suspends the Web Speech API when backgrounded regardless (a platform limitation, not fixable from this app without switching to real `<audio>`-based narration, which would need a backend).

**Theming**: CSS custom properties keyed off `data-theme` on `<html>` (`light`/`sepia`/`deep-dark`, applied by `useApplyTheme.ts`), defined in `src/index.css`. Two font families only, each with one job: `Spectral` (serif) for book content and titles, `IBM Plex Mono` for all UI chrome/labels/data. The `sans` reading-font option is the system UI stack, not a third custom font — deliberate, not an oversight. The landing page (`src/landing.css`) duplicates these same token values (deep-dark only, no theme switcher there) rather than importing `index.css`, so it doesn't accidentally couple to app-only styles added later.

**Keyboard shortcuts** (`useKeyboardShortcuts.ts`, all global via a single `window.keydown` listener): `Space` play/pause, `ArrowUp`/`ArrowDown` adjust WPM, `Shift+ArrowUp`/`Shift+ArrowDown` skip paragraph, `Esc` close command palette or toggle sidebar, `Cmd/Ctrl+K` toggle command palette. All except `Esc`/`Cmd+K` are ignored while focus is in an input/textarea/contentEditable.

**Mobile note**: sidebar is a full-overlay drawer below the `lg` breakpoint (defaults closed there via a `matchMedia` check on mount in `App.tsx`) and a static docked pane at `lg`+. `ControlBar` swaps the desktop tempo-ruler + `⌘K` button for a single compact `"{wpm} WPM"` button below `sm` that opens the same command palette — don't reintroduce desktop-only controls without a mobile equivalent or an explicit decision to omit one. Root layout uses `h-dvh`, not `h-screen` — `100vh` on mobile includes space behind the collapsible address bar, which pushes the bottom control bar below the fold until scrolled.

**Accessibility**: interactive elements use a shared `focus-visible:ring-2 focus-visible:ring-accent/60` pattern (no default browser outline is stripped, but this makes focus state on-brand and consistent). Paragraph blocks are keyboard-operable (`role="button"`, `tabIndex`, Enter/Space via `onKeyDown`), not just pointer-clickable.
